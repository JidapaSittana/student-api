## ส่วนที่ 1: ตารางสรุปผลการเปรียบเทียบ

| หัวข้อเปรียบเทียบ                                                     | REST API (`/api/v1/students/:id/full`)                 | GraphQL (`/graphql`)                                                                     |
| :-------------------------------------------------------------------- | :----------------------------------------------------- | :--------------------------------------------------------------------------------------- |
| **หัวข้อที่ 1: จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษา 1 คน** | ใช้ 1 คำขอ                                             | ใช้ 1 คำขอ                                                                               |
| **หัวข้อที่ 2: ฟิลด์ที่ได้รับเกินความจำเป็น (Over-fetching)**         | มีข้อมูลส่วนเกิน เช่น ฟิลด์ `email`, `phone`           | ไม่มีข้อมูลส่วนเกิน รับเฉพาะฟิลด์ `name` และ `courseName` ตามที่ระบุในคำขอ               |
| **หัวข้อที่ 3: จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษาทุกคน** | มากกว่า 1 คำขอ ถ้าใช้ Endpoint พื้นฐานต้องวนลูปดึงวิชา | ใช้ 1 คำขอ สั่งดึงความสัมพันธ์แบบ Nested Object ได้ข้อมูลครบถ้วนของทุกคนใน Request เดียว |

## ส่วนที่ 2: ตารางเปรียบเทียบรูปแบบการทำ CRUD ระหว่าง REST และ GraphQL

| การทำงาน                                                 | REST API                                                                                  | GraphQL                                                                                                    |
| :------------------------------------------------------- | :---------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------- |
| **การทำงานที่ 1: Create (สร้างข้อมูล)**                  | `POST /api/v1/students`                                                                   | `mutation { createStudent(input: CreateStudentInput!) }`                                                   |
| **การทำงานที่ 2: Read (อ่านข้อมูลรายการเดียว)**          | `GET /api/v1/students/:id`                                                                | `query { student(id: ID!) }`                                                                               |
| **การทำงานที่ 3: Read (อ่านหลายรายการ พร้อมกรอง/เรียง)** | ต้องส่งผ่าน Query String เช่น `?major=IT&sortBy=name` และเขียนลอจิกคัดกรองฝั่งหลังบ้านเอง | `query { students(major: "เทคโนโลยีสารสนเทศ", sortBy: "name") }` โดยกำหนดเป็น Argument ไว้ใน Schema ชัดเจน |
| **การทำงานที่ 4: Read (ข้อมูลสัมพันธ์ซ้อนกันหลายชั้น)**  | ต้องสร้าง Endpoint เฉพาะกิจเพิ่มเติม (เช่น `/full`) หรือทนรับการส่งข้อมูลดิบทั้งหมด       | ใช้ Nested Query โครงสร้างต้นไม้ดึงออบเจกต์สำเร็จใน 1 คำขอ                                                 |
| **การทำงานที่ 5: Update (แก้ไขข้อมูล)**                  | `PUT /api/v1/students/:id` หรือ `PATCH /api/v1/students/:id`                              | `mutation { updateStudent(id: ID!, input: UpdateStudentInput!) }`                                          |
| **การทำงานที่ 6: Delete (ลบข้อมูล)**                     | `DELETE /api/v1/students/:id`                                                             | `mutation { deleteStudent(id: ID!) }`                                                                      |

## ส่วนที่ 3: สรุปจากผลการทดลอง

- **การแก้ไขปัญหา Over-fetching:** จากที่เรียก REST Endpoint ในภาพที่ 1 จะเห็นฟิลด์ที่ไม่ได้ใช้งานหลุดมาด้วย แต่ถ้าใช้ GraphQL ในภาพที่ 2 ปัญหานี้ก็หมดไปเพราะเลือกเฉพาะฟิลด์ที่จำเป็น
- **การแก้ไขปัญหา Under-fetching:** ภาพที่ 3 แสดงว่า GraphQL ดึงข้อมูลความสัมพันธ์แบบ Nested ของนักศึกษาทุกคนกับรายวิชาได้จบในคำขอเดียว ต่างจาก REST ที่ต้องยิงกระจายหลาย Endpoint เพื่อเอาข้อมูลมารวมกัน
- **ความยืดหยุ่นฝั่ง Read vs Write:** ฝั่ง Read (Query) ของ GraphQL เปิดอิสระให้เลือกฟิลด์เอง แต่ฝั่ง Write (Mutation) ในภาพที่ 4 ต้องกำหนดอินพุตเพื่อให้เซิร์ฟเวอร์สามารถตรวจสอบความถูกต้องและความปลอดภัยของข้อมูลก่อนบันทึกจริง
