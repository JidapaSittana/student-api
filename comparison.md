# ตารางเปรียบเทียบ

---

## ตารางสรุปผลการเปรียบเทียบ

| หัวข้อเปรียบเทียบ                                        | REST API (`/api/v1/students/:id/full`)                 | GraphQL (`/graphql`)                                                                     |
| :------------------------------------------------------- | :----------------------------------------------------- | :--------------------------------------------------------------------------------------- |
| **จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษา 1 คน** | ใช้ 1 คำขอ                                             | ใช้ 1 คำขอ                                                                               |
| **ฟิลด์ที่ได้รับเกินความจำเป็น (Over-fetching)**         | มีข้อมูลส่วนเกิน เช่น ฟิลด์ `email`, `phone`           | ไม่มีข้อมูลส่วนเกิน รับเฉพาะฟิลด์ `name` และ `courseName` ตามที่ระบุในคำขอ               |
| **จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษาทุกคน** | มากกว่า 1 คำขอ ถ้าใช้ Endpoint พื้นฐานต้องวนลูปดึงวิชา | ใช้ 1 คำขอ สั่งดึงความสัมพันธ์แบบ Nested Object ได้ข้อมูลครบถ้วนของทุกคนใน Request เดียว |

---

## สรุปจากผลการทดลอง

- **การแก้ไขปัญหา Over-fetching:** จากที่เรียก REST Endpoint ภาพที่1 จะเห็นฟิลด์ที่ไม่ได้ใช้งานหลุดมาด้วย แต่ถ้าใช้ GraphQL ภาพที่2 ปัญหานี้ก็หมดไปเพราะเลือกเฉพาะฟิลด์ที่จำเป็น
- **การแก้ไขปัญหา Under-fetching:** ภาพที่3 แสดงว่า GraphQL ดึงข้อมูลความสัมพันธ์แบบ Nested ของนักศึกษาทุกคนกับรายวิชาได้จบในคำขอเดียว ต่างจาก REST ที่ต้องยิงกระจายหลาย Endpoint เพื่อเอาข้อมูลมารวมกัน
- **ความยืดหยุ่นฝั่ง Read กับ Write:** ฝั่ง Read ของ GraphQL เปิดอิสระให้เลือกฟิลด์เอง แต่ฝั่ง Write ภาพที่4 ต้องกำหนดอินพุตเพื่อให้เซิร์ฟเวอร์ตรวจสอบความถูกต้องและความปลอดภัยของข้อมูลก่อนบันทึกจริง
